Introduction: From Evidence to Narrative

You‚Äôve completed the HCI design cycle: audit ‚Üí pilot ‚Üí analyse ‚Üí redesign ‚Üí verify. Today you present that work to peers.

This is not ‚Äúshow and tell‚Äù. Studio critique is:

  • Evidence-led: Every claim backed by data, screenshots, transcripts
  • Reflective: Acknowledge limitations, trade-offs, remaining issues
  • Actionable: Invite specific feedback, identify next steps

Why this matters:

  • Professional practice: Design reviews are standard in industry (Google, GOV.UK, Microsoft all use critique protocols)
  • Accreditation: External panels assess evidence chains‚Äîthis is rehearsal
  • Peer learning: Seeing 5-6 different solutions to same problem builds design vocabulary

Format: 15 min per team (5 min demo, 3 min metrics, 2 min accessibility proof, 5 min Q&A). Tight timing forces clarity.

Key Concepts

Design Critique

Design Critique (Glossary)

Structured peer review of design work. Not personal feedback‚Äîfocuses on work, evidence, impact.

Characteristics

  • Specific: ‚ÄúError summary not keyboard-focusable‚Äù (good) vs ‚Äúerrors bad‚Äù (vague)
  • Constructive: Suggest alternatives, not just problems
  • Evidence-based: Reference data, WCAG, observations (not opinions)
  • Respectful: Assume positive intent, acknowledge constraints

Format (this module)

  • Presenter shows work + evidence
  • Audience asks clarifying questions
  • Audience offers critique: ‚ÄúI noticed X, have you considered Y?‚Äù
  • Presenter notes feedback (doesn‚Äôt defend in real-time)

HCI Connection: Critique improves design through external perspective‚Äîyou‚Äôre too close to your work to see all issues.

Not a critique

  • ‚ÄúI don‚Äôt like the colour‚Äù (subjective preference without rationale)
  • ‚ÄúThis sucks‚Äù (not constructive)
  • ‚ÄúYou should have‚Ä¶‚Äù (hindsight, not helpful)

Good critique

  • ‚ÄúThe error message says ‚ÄòTitle required‚Äô but doesn‚Äôt explain the max length constraint. Pilot P3 hit this‚Äîcould you add that info?‚Äù

Evidence-Led Presentation

Evidence-Led Presentation (Glossary)

Presentation style where every claim is backed by artifacts (data, screenshots, quotes).

Structure

  1. Problem: ‚ÄúT2 edit had 67% completion (0% no-JS)‚Äù ‚Üê metric
  2. Root cause: ‚ÄúValidation errors not announced to SR‚Äù ‚Üê pilot quote + WCAG reference
  3. Fix: ‚ÄúAdded role=alert‚Äù ‚Üê code diff
  4. Verification: ‚ÄúPost-change: 100% completion‚Äù ‚Üê after-metrics + screenshot

What NOT to do

  • ‚ÄúWe made it better‚Äù (vague‚Äîbetter how?)
  • ‚ÄúUsers struggled‚Äù (who? how many? with what?)
  • ‚ÄúFixed accessibility‚Äù (which criteria? verified how?)

HCI Connection: Evidence separates opinion from fact. Stakeholders can question interpretation, but not the data.

Academic context: External examiners check evidence chains. Missing evidence ‚Üí lower marks, even if work was good.

Accessibility Demonstration

Accessibility Demonstration (Glossary)

Live proof that inclusive design works. Show, don‚Äôt tell.

Examples

  • Keyboard: Tab through form ‚Üí submit error ‚Üí Tab to error link ‚Üí press Enter ‚Üí focus lands on input
  • Screen reader: Turn on NVDA/Orca ‚Üí navigate to form ‚Üí submit error ‚Üí SR announces ‚ÄúAlert. Title is required‚Äù
  • No-JS: Disable JavaScript ‚Üí submit form ‚Üí error summary appears and focused

Why live demo? Screenshots can be faked. Live demo proves system works right now.

Backup plan: If live demo fails (demo gremlins), have video recording or annotated screenshots ready.

What to show

  • Before state (broken or suboptimal)
  • After state (fixed)
  • Evidence it meets WCAG (checklist row, transcript)

Common mistakes

  • Only showing HTMX path (ignoring no-JS)
  • Not narrating what‚Äôs happening (‚ÄúI‚Äôm pressing Tab, now I‚Äôm on the error link‚Ä¶‚Äù)
  • Assuming audience sees what you see (make focus indicators obvious, zoom in if needed)

Constructive Feedback

Constructive Feedback (Glossary)

Feedback that helps recipient improve. Has three parts: observation + impact + suggestion.

Formula

  1. Observation: ‚ÄúI noticed X‚Äù (specific, factual)
  2. Impact: ‚ÄúThis might affect Y‚Äù (consequence, not judgment)
  3. Suggestion: ‚ÄúHave you considered Z?‚Äù (alternative, not mandate)

Example (good)

  • Observation: ‚ÄúYour error summary uses aria-live=assertive, which I saw announced in your SR demo‚Äù
  • Impact: ‚ÄúIn my testing, assertive interrupted mid-sentence when I was reading other content‚Äù
  • Suggestion: ‚ÄúHave you tested with aria-live=polite to see if it‚Äôs less disruptive? Might be a good trade-off if errors aren‚Äôt time-critical.‚Äù

Example (bad)

  • ‚ÄúYou should use polite not assertive‚Äù ‚Üê prescriptive, no context

Receiving feedback

  • Listen: Don‚Äôt defend immediately (note it, reflect later)
  • Clarify: ‚ÄúCan you show me where you saw that?‚Äù
  • Thank: Even if you disagree, feedback is effort
