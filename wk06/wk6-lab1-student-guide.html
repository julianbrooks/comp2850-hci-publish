Understanding the Code (15 minutes)

1. Server-First Pattern

Open: src/main/kotlin/routes/TaskRoutes.kt

Key function: get("/tasks") (lines 67-77)

What it does:

  1. Gets all tasks from repository
  2. Creates a model (map of data)
  3. Renders tasks/index.peb template with data
  4. Sends complete HTML to browser

Why this matters: The server sends ready-to-display HTML. If JavaScript fails, the page still works.

2. Dual-Mode Architecture

Still in: TaskRoutes.kt

Key function: post("/tasks") (lines 83-122)

The pattern:

if (call.isHtmx()) {
    // HTMX path: return HTML fragment
    return@post call.respondText(fragment + status, ...)
}

// No-JS path: redirect to GET /tasks
call.respond(HttpStatusCode.SeeOther)

What this means:

  • With JavaScript: HTMX intercepts form submit, server returns just the new task HTML, HTMX adds it to the list (smooth, no page reload)
  • Without JavaScript: Form submits normally, server redirects browser back to /tasks, full page reloads (works but slower)

Why both?: Progressive enhancement - baseline functionality for everyone, enhanced experience for capable browsers.

3. HTMX Attributes

Open: src/main/resources/templates/tasks/index.peb

Find the form (around line 9-18):

<form action="/tasks" method="post"
      hx-post="/tasks"
      hx-target="#task-list"
      hx-swap="beforeend">

What each attribute does:

  • action="/tasks" method="post" - Baseline HTML (works without JS)
  • hx-post="/tasks" - HTMX intercepts, does AJAX instead
  • hx-target="#task-list" - Where to put the response
  • hx-swap="beforeend" - How to insert it (append to end of list)

The magic: If JavaScript is disabled, hx-* attributes are ignored and the form uses action instead. Both paths work!

4. Out-of-Band Status Updates

Back in: TaskRoutes.kt line 114

val status = """<div id="status" hx-swap-oob="true">Task "${task.title}" added successfully.</div>"""

What hx-swap-oob="true" does:

  • “OOB” = Out Of Band
  • Response includes this <div> even though it’s not in the hx-target
  • HTMX finds the element with id="status" anywhere on the page and updates it
  • Screen readers announce the change (because the div has role="status" in base layout)

Why this matters: Visual feedback + screen reader announcement with one server response.

Verification Checklist

Test that everything works correctly:

Dual-Mode Testing

  • Run the server: ./gradlew run → visit http://localhost:8080/tasks
  • Add task (with JS): Type “Buy milk” → click Add Task → task appears without page reload
  • Delete task (with JS): Click Delete → task disappears without page reload
  • Disable JavaScript: DevTools → Settings → Disable JavaScript → refresh page
  • Add task (no JS): Type “Pay bills” → click Add Task → page reloads, task appears
  • Delete task (no JS): Click Delete → page reloads, task gone

Expected: All 6 tests pass. Features work identically with/without JavaScript.

Accessibility Testing

  • Keyboard navigation:

    • Press Tab repeatedly → focus moves through: skip link → title input → Add button → delete buttons → footer
    • All focus indicators visible (blue outline)
    • Press Enter on Add button → form submits

  • Skip link:

    • Press Tab once → “Skip to main content” appears
    • Press Enter → focus jumps to main content (skips header)

  • Screen reader (if available - NVDA/ORCA/VoiceOver):

    • Navigate to form → hears “Title, edit text, Keep it short and specific”
    • Add task → hears “Task ‘Buy milk’ added successfully”
    • Navigate to delete button → hears “Delete task: Buy milk, button”

  • ARIA live region:

    • Open DevTools → Elements tab → find <div id="status">
    • Add a task → watch the status div text change
    • Confirm it has role="status" and aria-live="polite"

Browser History

  • Add task → URL stays /tasks (not /tasks? or changed)
  • Press back button → goes to previous page (not re-submits form)
  • Refresh page → shows current task list (not “confirm resubmission” dialog)

Why this works: Post-Redirect-Get (PRG) pattern. After POST, server sends 303 See Other redirect to GET /tasks.

Common Issues & Fixes

“Cannot find TaskRepository”

Problem: Missing import or file not created Fix: Check src/main/kotlin/data/TaskRepository.kt exists. If not, copy from lab instructions Activity 3 Step 1.

“Tasks don’t appear after adding”

Problem: HTMX target mismatch Fix: In index.peb, confirm <ul id="task-list"> exists. HTMX needs this ID to know where to insert.

“Page reloads even with JavaScript enabled”

Problem: HTMX library not loaded Fix: Check browser console for errors. Confirm static/js/htmx-1.9.12.min.js exists and is linked in base.peb.

“Status message not announced by screen reader”

Problem: Live region not configured correctly Fix: Check _layout/base.peb has <div id="status" role="status" aria-live="polite"> in header.
