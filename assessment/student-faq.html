Academic Year: 2025-26 Assessment: Combined Evaluation & Redesign Portfolio Timeline: Weeks 9-11

General Questions

Q: What am I submitting?

A: One complete portfolio with these files:

  1. submission-template.md ‚Äî Single markdown file (all sections filled in)
  2. evidence/ folder ‚Äî Screenshots (no PII), pilot notes

Total word count: ~100-200 words (evidence-based, not essay!)

Q: When is the deadline?

A: End of Week 10 (submit via Gradescope)

Timeline:

  • Week 9: Run peer pilots, collect metrics.csv
  • Week 10: Implement fixes, regression test, peer review
  • Week 11: Incorporate feedback, final polish, submit

Q: Is this pass/fail?

A: Yes, pass/fail outcome (40% threshold), but you also receive a 0-100 mark for formative feedback showing quality level:

  • 70-100: First (excellent)
  • 60-69: 2:1 (very good)
  • 50-59: 2:2 (good)
  • 40-49: 3rd (minimum pass)
  • 0-39: Fail

Participants & Pilots

Q: How many participants do I need?

A: Minimum n=2, ideal n=4

Why n=2 minimum? Shows pattern (not anecdotal evidence) Why not require n=4? Flexible for scheduling/availability

Variant requirement: At least 1 non-standard variant (keyboard-only OR screen reader OR no-JS)

Q: Can I pilot with friends?

A: Yes, if they consent! Peer pilots are ideal. Just document consent process (read script verbatim, record timestamp).

Q: What if I can‚Äôt find participants?

A:

  • Post in module discussion forum (‚Äúseeking pilot participants‚Äù)
  • Coordinate with lab peers
  • Ask TAs to help connect students
  • Pilots can happen outside lab time (just by Week 9 deadline)

Q: Can I test on the same participants for re-pilots (verification)?

A: Yes, if they consent. They‚Äôre now ‚Äúexpert users‚Äù ‚Äî document this limitation in verification.csv notes.

Q: What if I only get n=1 participant?

A: That‚Äôs below minimum. Run at least 1 more pilot. n=2 is required to show pattern. If genuinely stuck despite effort, talk to TA about extension.

What to Submit

Q: Do I need to write essays?

A: No! This is evidence-based. Total prose: ~100-200 words (task scenarios, impact statements, alt text). Everything else is CSV, tables, code diffs, checklists.

Example:

  • ‚ùå Essay: ‚ÄúI improved accessibility by conducting rigorous WCAG evaluations and implementing comprehensive ARIA enhancements across multiple interface components‚Ä¶‚Äù
  • ‚úÖ Evidence: findings-table.csv ‚Üí implementation-diffs.md ‚Üí verification.csv (data ‚Üí fix ‚Üí verification)

Q: What files do I submit?

A:

  1. submission-template.md (completed) ‚Äî One file with all sections
  2. evidence/ folder containing:
    • Screenshots (before/after, no PII)
    • Pilot notes (P1-notes.md, P2-notes.md, etc.)

metrics.csv can be pasted into submission-template.md Section 3 OR attached separately.

Q: How do I scrub PII?

A: Before submission:

  • Crop screenshots to show only relevant UI (no browser bookmarks, tabs, usernames)
  • Use anonymous session IDs (P1_xxxx format, not real names)
  • Blur any visible names, emails, student IDs
  • Check pilot notes use participant codes (P1, P2) not names

Critical: Severe PII violations (names/emails visible) = automatic fail.

Evidence & Data

Q: What if my data is ‚Äútoo messy‚Äù (lots of errors)?

A: That‚Äôs good data! Real evaluation shows errors, confusion, failures. Report honestly.

Fabricated data (100% success, no errors, unrealistic times) is a fail trigger. Authentic data (even messy) is valued.

Q: What if my fix didn‚Äôt improve metrics?

A: Document honestly! Analyse why:

  • Small sample size (n=2 may not show clear improvement)
  • Wrong fix (addressed symptom not root cause)
  • Need more iteration

In verification.csv notes, write: ‚ÄúFix did not improve metrics, likely because [small n / wrong diagnosis / needs different approach].‚Äù

Honest reporting is professional practice (LO12). Fabricating improvement is fail.

Q: My metrics.csv is huge (200+ rows). Do I submit it all?

A: Yes! Paste all rows in submission-template.md Section 3 OR attach as separate file. Staff use this to verify patterns.

If truly massive (500+ rows), attach as separate file and note in template: ‚ÄúSee attached metrics.csv‚Äù.

Technical Questions

  2. No rows appearing: Check you‚Äôre calling Logger.log() in route handlers.
  3. JS mode always ‚Äúon‚Äù: Check HX-Request header detection (call.request.headers["HX-Request"] == "true").
  4. Doesn‚Äôt work in no-JS mode: Logger.kt is server-side (should work). Check routing ‚Äî are no-JS requests reaching your POST handler?

Grading & Quality

Q: What makes a First (70+)?

Characteristics:

  • All evidence chains rigorous and complete (can trace 3+ findings from data ‚Üí fix ‚Üí verification)
  • WCAG compliance thorough (20/20 regression checks, no Level A violations)
  • Professional documentation (no PII, honest reporting of limitations)
  • Code fixes target root causes, well-integrated
  • Before/after metrics show measurable improvement (or honest analysis)

Example: Finding #1 traceable: metrics.csv L47-49 ‚Üí pilot notes P2 14:23 ‚Üí findings table ‚Üí Fix #1 diffs ‚Üí verification.csv 3.3.1 pass ‚Üí before/after 0% ‚Üí 100%.

Q: What‚Äôs the minimum to pass (40)?

Minimum requires:

  • n=2 participants (verified in metrics.csv)
  • 3 findings with some data sources (even if vague)
  • 1 fix with code diff shown
  • Partial regression (at least 10/20 checks, ideally 20)
  • No fabrication, no severe PII violations

Q: Do I need to fix ALL my findings?

A: No! Focus on 1-3 high-priority issues. Quality over quantity.

Good approach:

  • Prioritise using (Impact + Inclusion) - Effort
  • Select top 2-3 fixes
  • Verify thoroughly

Bad approach:

  • Try to fix 8 issues
  • Half-finish all of them
  • Poor verification

Q: Can I introduce new features?

A: No. This is redesign, not new feature development. Fix existing usability issues only.

Peer Review (Week 10 Lab 2)

Q: What is peer review?

A: Structured in-lab session (15 mins) where groups of 4 review each other‚Äôs portfolios using a checklist. Focus: Are files present? n=2+? 3+ findings? PII scrubbed?

Not graded. Helps you catch gaps before final submission.

Q: What if my peer gives me conflicting feedback?

A: Peer review is formative (not binding). Consider their perspective ‚Äî if they couldn‚Äôt find your evidence in 2 minutes, staff might struggle too. Make links clearer.

Q: What if I disagree with my peer‚Äôs assessment?

A: That‚Äôs fine. Use feedback to improve, but trust your own judgment. If peer says ‚Äúmissing files‚Äù but you have them, double-check file names match submission-template.md references.

Common Pitfalls

Pitfall 1: Trying to fix everything

Problem: 8 issues identified, try to fix all, run out of time Solution: Prioritise 2-3 high-priority fixes using (Impact + Inclusion) - Effort

Pitfall 2: Vague data sources

Problem: findings-table.csv says ‚Äúpilot observations‚Äù instead of specific sources Solution: Use metrics.csv line numbers (e.g., ‚ÄúL47-49‚Äù) or pilot notes timestamps (e.g., ‚ÄúP2 notes 14:23‚Äù)

Pitfall 3: No before/after comparison

Problem: ‚ÄúI fixed it‚Äù (no metrics showing improvement) Solution: Run 1-2 re-pilots after fixes, compare to baseline (verification.csv Part B)

Pitfall 4: PII in screenshots

Problem: Participant names, emails visible in evidence Solution: Crop/blur before submission. Check twice! Automatic fail if not corrected.

Pitfall 5: Incomplete regression

Problem: Only 5/20 WCAG checks completed Solution: Complete all 20 checks using Assistive Testing Checklist. If time-limited, minimum ~10 checks for pass.

Specific Scenarios

Q: My participant completed all tasks perfectly (no errors). Is that OK?

A: Unusual but possible (expert user, very simple tasks). Make sure:

  • You have pilot notes (quotes, observations, timestamps)
  • You can explain their session if asked
  • You‚Äôre testing appropriate participants (not HCI experts who know the answers)

If data seems ‚Äútoo perfect‚Äù (100% success, all times <5s), that‚Äôs a red flag for fabrication.

Q: I have 3 findings but they‚Äôre all minor (e.g., button label typos). Will I fail?

A: Depends. If all findings are trivial and don‚Äôt address accessibility/usability, marks will be lower (50-60 range). Better approach:

  • Use WCAG regression checklist to find violations
  • Review pilot notes for participant confusion/errors
  • Look for patterns in metrics.csv (validation errors, high times)

Q: My re-pilot metrics are worse than baseline. Failed fix?

A: Potentially, but document honestly! In verification.csv notes:

  • ‚ÄúMetrics worse ‚Äî possible regression introduced by fix‚Äù
  • ‚ÄúNeed to investigate unintended side effects‚Äù
  • ‚ÄúMay require different approach‚Äù

Honest analysis valued. Fabricating improvement is fail.

Q: I only implemented 1 fix. Is that enough?

A: Yes, if:

  • Fix is substantial (addresses major usability/accessibility issue)
  • Thoroughly verified (regression + before/after)
  • Well-documented (code diff + rationale)

1 fix done well > 3 fixes half-finished.

After Submission

Q: When do I get my mark?

A: Marks released according to university assessment timeline (typically 3-4 weeks after deadline). Check VLE announcements.

  • Evidence of bias

Not grounds for appeal: ‚ÄúI think I deserved higher‚Äù or ‚ÄúI worked really hard‚Äù. Marks based on evidence-based rubric.
