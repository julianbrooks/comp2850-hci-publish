COMP2850 • Human-Computer Interaction

Marking Rubric

Introduction

This rubric shows exactly how tasks are marked. We assess HCI process competence, not code quality or writing style.

Key principles:

  • Evidence over eloquence: Tables, metrics, and code diffs matter more than prose
  • Process over product: We mark your method (protocol, testing, analysis), not your app
  • Honesty over perfection: Reporting limitations honestly scores higher than hiding problems
  • Rigour over scope: One fix done thoroughly beats three fixes done poorly

Task 1: Evaluation & Findings

Total: 100 marks (see Minerva for weighting within overall COMP2850 grade)

Evidence Quality (30 marks)

Excellent (21-30 marks)

  • ✅ Complete quantitative data (04-results.csv with all columns, n≥4 participants)
  • ✅ Rich qualitative data (pilot notes with timestamps, direct quotes, specific observations)
  • ✅ Traceable evidence chains (every finding links to data: file path, line number, session ID)
  • ✅ No PII in any file (screenshots cropped, session IDs anonymous)
  • ✅ Multiple data sources converge (metrics + quotes + screenshots all tell same story)

Good (18-20 marks)

  • ✅ Complete quantitative data (n=4, all columns present)
  • ✅ Adequate qualitative data (pilot notes present, some quotes)
  • ✅ Most findings have evidence chains (1-2 minor gaps)
  • ✅ No PII violations
  • ⚠️ Evidence chains occasionally vague (“some participants struggled” without specifics)

Satisfactory (15-17 marks)

  • ✅ Quantitative data present (n=4 minimum met)
  • ⚠️ Thin qualitative data (pilot notes brief, few quotes)
  • ⚠️ Evidence chains incomplete (findings asserted without clear data links)
  • ✅ No PII violations
  • ⚠️ Data sources sometimes contradict (metrics say 100% completion, notes say “struggled”)

Needs Work (12-14 marks)

  • ⚠️ Incomplete quantitative data (missing sessions, columns incomplete)
  • ⚠️ Minimal qualitative data (pilot notes cursory or missing)
  • ❌ Evidence chains absent (claims without data)
  • ⚠️ Minor PII issues (session IDs identifiable, screenshots not scrubbed)

Fail (0-11 marks)

  • ❌ Fabricated data (markers can detect unrealistic patterns)
  • ❌ n<4 participants (minimum not met)
  • ❌ Major PII violations (names, emails visible)
  • ❌ No evidence chains (entire findings section unsupported)

Method Rigour (25 marks)

Excellent (18-25 marks)

  • ✅ Protocol includes consent script, right to withdraw, ethical safeguards
  • ✅ Tasks realistic, measurable, independent (clear success criteria)
  • ✅ Appropriate variants tested (keyboard, no-JS, screen reader)
  • ✅ Link to Week 6 needs-finding demonstrated (LO2: job stories → task design)
  • ✅ Pilot execution follows protocol (facilitator neutral, observer notes structured)

Good (15-17 marks)

  • ✅ Protocol includes consent and ethical safeguards
  • ✅ Tasks measurable and realistic
  • ✅ Minimum 3 variants tested
  • ⚠️ Needs-finding link present but superficial (brief mention, not integrated)
  • ✅ Protocol mostly followed

Satisfactory (13-14 marks)

  • ✅ Basic protocol present (consent mentioned)
  • ⚠️ Tasks vague (success criteria unclear)
  • ✅ 2-3 variants tested (minimum met)
  • ⚠️ Needs-finding link weak or missing
  • ⚠️ Protocol execution inconsistent (some pilots rushed)

Needs Work (10-12 marks)

  • ⚠️ Protocol incomplete (consent unclear, no right to withdraw)
  • ❌ Tasks poorly designed (not measurable or independent)
  • ⚠️ Limited variants (only standard, no keyboard/no-JS)
  • ❌ No link to needs-finding (LO2 not addressed)

Fail (0-9 marks)

  • ❌ No consent protocol (ethical violation)
  • ❌ Tasks not suitable for evaluation
  • ❌ Only 1-2 pilots conducted
  • ❌ Protocol not followed (leading questions, no observer notes)

Analysis Depth (25 marks)

Excellent (18-25 marks)

  • ✅ Quantitative analysis correct (median/MAD calculated, error rates accurate)
  • ✅ Qualitative synthesis identifies patterns across participants
  • ✅ WCAG violations cited with specific criteria (4.1.3, 3.3.1, etc.)
  • ✅ Inclusion impact articulated (who is excluded, how, why)
  • ✅ Prioritisation uses formula: (Impact + Inclusion) – Effort with justification
  • ✅ Limitations acknowledged (sample size, diversity, context)

Good (15-17 marks)

  • ✅ Quantitative analysis mostly correct (minor calculation errors)
  • ✅ Qualitative synthesis present, some patterns identified
  • ✅ WCAG violations cited (may lack specific criteria numbers)
  • ✅ Inclusion impact mentioned (SR users, keyboard users)
  • ⚠️ Prioritisation present but formula not applied consistently
  • ⚠️ Limitations briefly mentioned

Satisfactory (13-14 marks)

  • ⚠️ Quantitative analysis basic (completion rates calculated, but no MAD/median)
  • ⚠️ Qualitative synthesis superficial (lists issues without patterns)
  • ⚠️ WCAG references generic (“accessibility issues” not specific criteria)
  • ⚠️ Inclusion impact vague (“helps disabled people”)
  • ⚠️ Prioritisation arbitrary (no formula, unclear rationale)
  • ❌ Limitations not discussed

Needs Work (10-12 marks)

  • ❌ Quantitative analysis incorrect or missing
  • ❌ Qualitative synthesis absent (just lists observations)
  • ❌ No WCAG references
  • ❌ Inclusion impact not discussed
  • ❌ No prioritisation

Fail (0-9 marks)

  • ❌ No analysis (raw data dumped without interpretation)
  • ❌ Findings fabricated or unsupported
  • ❌ No evidence of understanding WCAG or accessibility concepts

Professional Presentation (20 marks)

Excellent (15-20 marks)

  • ✅ Clear, well-structured documentation (all 6 files present, organised)
  • ✅ Honest reporting (limitations acknowledged, unexpected results explained)
  • ✅ Complete submission (directory structure matches specification)
  • ✅ UK spelling and grammar throughout
  • ✅ Proper Markdown formatting (tables render, headings hierarchical)
  • ✅ Evidence organised (screenshots annotated, pilot notes templated)

Good (12-14 marks)

  • ✅ Documentation clear and complete
  • ✅ Honest reporting
  • ✅ Submission complete
  • ⚠️ Minor spelling/grammar issues (1-2 Americanisms)
  • ⚠️ Formatting mostly correct (1-2 tables malformed)
  • ✅ Evidence organised

Satisfactory (10-11 marks)

  • ⚠️ Documentation adequate but could be clearer
  • ✅ Reporting mostly honest
  • ⚠️ 1 file missing or incomplete
  • ⚠️ Spelling/grammar errors present
  • ⚠️ Formatting issues (some tables don’t render)
  • ⚠️ Evidence somewhat disorganised

Needs Work (8-9 marks)

  • ❌ Documentation unclear or disorganised
  • ⚠️ Reporting hides problems (only positive results shown)
  • ❌ 2+ files missing
  • ❌ Poor spelling/grammar
  • ❌ Formatting broken (Markdown errors)

Fail (0-7 marks)

  • ❌ Submission incomplete (half the files missing)
  • ❌ Dishonest reporting (fabricates data, hides failures)
  • ❌ Plagiarism or academic integrity violation

Task 2: Redesign & Verification

Total: 100 marks (see Minerva for weighting within overall COMP2850 grade)

Problem Clarity (20 marks)

Excellent (15-20 marks)

  • ✅ Problem statement grounded in Task 1 data (specific findings cited: file + section)
  • ✅ Measurable goals with targets (completion rate, error rate, median time, SR accessibility)
  • ✅ WCAG violations referenced with specific criteria (4.1.3, 3.3.1, etc.)
  • ✅ Prioritisation justified using (Impact + Inclusion) – Effort formula

Good (12-14 marks)

  • ✅ Problem grounded in Task 1 (findings cited)
  • ✅ Goals measurable
  • ⚠️ WCAG references generic (criteria numbers missing)
  • ⚠️ Prioritisation mentioned but formula not applied

Satisfactory (10-11 marks)

  • ⚠️ Problem references Task 1 (vague link)
  • ⚠️ Goals present but not fully measurable
  • ❌ No WCAG references
  • ❌ Prioritisation arbitrary

Needs Work / Fail (0-9 marks)

  • ❌ Problem not grounded in Task 1
  • ❌ Goals missing or unmeasurable
  • ❌ No prioritisation

Fix Effectiveness (30 marks)

Excellent (22-30 marks)

  • ✅ WCAG 2.2 AA compliance verified (regression checklist complete, axe DevTools clean)
  • ✅ Measurable improvement demonstrated (before/after metrics show gains)
  • ✅ No-JS parity maintained (traditional path tested, works identically)
  • ✅ No regressions introduced (30-check checklist all PASS)
  • ✅ Fixes target high inclusion impact (WCAG violations, SR/keyboard barriers)

Good (18-21 marks)

  • ✅ WCAG AA compliance verified (1-2 minor issues remain)
  • ✅ Improvement demonstrated
  • ✅ No-JS parity maintained
  • ⚠️ 1-2 minor regressions (documented and explained)
  • ✅ Fixes target inclusion

Satisfactory (15-17 marks)

  • ⚠️ WCAG AA mostly compliant (some gaps)
  • ⚠️ Modest improvement (metrics improve slightly)
  • ⚠️ No-JS parity mostly maintained (1 minor break)
  • ⚠️ Some regressions not caught
  • ⚠️ Fixes address some accessibility issues

Needs Work (12-14 marks)

  • ❌ WCAG violations remain (regression checklist incomplete)
  • ❌ No measurable improvement (metrics unchanged or worse)
  • ❌ No-JS parity broken
  • ❌ Multiple regressions introduced

Fail (0-11 marks)

  • ❌ Fixes don’t work or introduce major accessibility issues
  • ❌ No regression testing performed
  • ❌ No-JS completely broken
  • ❌ Academic integrity issues (fixes copied without understanding)

Societal Impact Analysis (LO10) (20 marks)

Excellent (15-20 marks)

  • ✅ Systemic barriers identified (beyond individual inconvenience: employment, education, civic participation)
  • ✅ Exclusion patterns analysed (who is excluded, how, why—not just “disabled people”)
  • ✅ Universal design distinguished from accommodation (fix benefits all users, not just workaround)
  • ✅ Ethical considerations addressed (privacy, autonomy, dignity)
  • ✅ Long-term implications discussed (if pattern adopted widely, societal change)
  • ✅ Critical depth: 300-500 words of analysis (not description)

Good (12-14 marks)

  • ✅ Systemic barriers mentioned (some analysis beyond individual)
  • ✅ Exclusion patterns identified (specific groups)
  • ⚠️ Universal design vs accommodation touched on (not deeply explored)
  • ⚠️ Ethical considerations brief
  • ⚠️ Long-term implications mentioned (not analysed)
  • ⚠️ 200-300 words

Satisfactory (10-11 marks)

  • ⚠️ Barriers mentioned but not systemic (individual inconvenience)
  • ⚠️ Exclusion generic (“disabled people” without specifics)
  • ❌ No distinction between universal design and accommodation
  • ❌ Ethics not discussed
  • ❌ No long-term analysis
  • ⚠️ 150-250 words

Needs Work (8-9 marks)

  • ❌ Superficial (“helps disabled people” without analysis)
  • ❌ No systemic thinking
  • ❌ <150 words or missing

Fail (0-7 marks)

  • ❌ Section missing or completely inadequate
  • ❌ Shows no understanding of LO10

Evidence Chains (20 marks)

Excellent (15-20 marks)

  • ✅ Every fix links to Task 1 finding (file + section reference)
  • ✅ Before/after metrics comparison complete (pre/analysis.csv vs post/postchange.csv)
  • ✅ Code diffs complete for all changes (before/after snippets with annotations)
  • ✅ Screenshots annotated (alt text + context)
  • ✅ Regression tests documented (30 checks complete, notes for each)

Good (12-14 marks)

  • ✅ Most fixes link to Task 1 (1-2 minor gaps)
  • ✅ Metrics comparison complete
  • ⚠️ Code diffs mostly complete (1-2 changes lack snippets)
  • ⚠️ Screenshots present (annotations brief)
  • ✅ Regression tests complete

Satisfactory (10-11 marks)

  • ⚠️ Some fixes link to Task 1 (gaps present)
  • ⚠️ Metrics comparison incomplete (only 1-2 metrics compared)
  • ⚠️ Code diffs superficial (claims changed but minimal evidence)
  • ⚠️ Screenshots present but not annotated
  • ⚠️ Regression checklist incomplete (blanks, N/As)

Needs Work / Fail (0-9 marks)

  • ❌ No link to Task 1
  • ❌ No before/after comparison
  • ❌ No code diffs
  • ❌ Regression checklist absent or mostly blank

Professional Presentation (10 marks)

Excellent (8-10 marks)

  • ✅ Clear, well-structured documentation (all 6 files + directories)
  • ✅ Honest reporting (if metrics didn’t improve, explains why)
  • ✅ Complete submission (matches specification exactly)
  • ✅ UK spelling and grammar
  • ✅ Proper Markdown formatting

Good (6-7 marks)

  • ✅ Documentation clear
  • ✅ Honest reporting
  • ⚠️ 1 minor file/directory missing
  • ⚠️ Minor spelling/grammar issues
  • ⚠️ Minor formatting issues

Satisfactory / Needs Work (4-5 marks)

  • ⚠️ Documentation adequate
  • ⚠️ Reporting hides some issues
  • ❌ 2+ files missing
  • ❌ Spelling/grammar errors

Fail (0-3 marks)

  • ❌ Submission incomplete
  • ❌ Dishonest reporting
  • ❌ Plagiarism or academic integrity violation

Academic Integrity

Automatic Fail Triggers

The following result in immediate zero marks and referral to Academic Integrity Office:

❌ Fabricated pilot data (markers can detect unrealistic patterns: too perfect, identical times, no variation)

❌ Plagiarism (code/analysis copied from peers or internet without attribution)

❌ Contract cheating (someone else did the work)

❌ Falsified evidence (screenshots doctored, metrics invented, consent not obtained)

❌ No actual pilots conducted (claims 4 pilots but only did 1-2, or none)

Serious Penalties

The following result in major mark deductions (30-50% penalty):

⚠️ Major PII violations (participant names/emails visible, identifiable session IDs)

⚠️ No consent obtained (pilots conducted without ethical protocol)

⚠️ Significant plagiarism (substantial sections copied, inadequate attribution)

Common Misconceptions

❌ “I need perfect code to get high marks”

No. We don’t mark code quality. We mark HCI process. You can have bugs in your app and still get Excellent if your evaluation/redesign method is rigorous.

❌ “I need to write 3000 words to get good marks”

No. We prefer concise tables and metrics over verbose prose. A 1200-word findings doc with strong evidence beats a 3000-word essay with weak data.

❌ “If my redesign doesn’t improve metrics, I’ll fail”

No. We mark honest analysis. If metrics don’t improve, explain why (small n? wrong fix? need more iteration?). Shows LO12 (professionalism).

❌ “I need to fix all issues found in Task 1”

No. We mark prioritisation and thoroughness. 2 fixes done rigorously (WCAG verified, regression tested, re-piloted) beats 10 fixes done superficially.

❌ “I can’t get Excellent if I find problems in my work”

No. Reporting limitations honestly is a mark of professionalism (LO12). Hiding problems loses marks; acknowledging them gains marks.

How to Self-Assess Before Submission

Task 1 Checklist

  • Evidence: Every finding has file path, line number, or session ID reference?
  • n=4 minimum: At least 4 participants piloted?
  • Variants: At least 3 variants tested (standard, keyboard, no-JS)?
  • Needs-finding link: Connected evaluation to Week 6 job stories (LO2)?
  • Consent: Protocol includes consent script and right to withdraw?
  • No PII: All screenshots cropped, session IDs anonymous?
  • WCAG: Specific criteria cited (4.1.3, 3.3.1, etc.)?
  • Honest: Limitations acknowledged, unexpected results explained?

Task 2 Checklist

  • Grounded: Problem links to specific Task 1 findings (file + section)?
  • WCAG AA: Regression checklist complete (all 30 checks PASS)?
  • Before/after: Metrics comparison shows improvement?
  • No-JS parity: Traditional path tested and works?
  • Societal impact: 300-500 words analysing systemic barriers (LO10)?
  • Code diffs: Before/after snippets for all changes?
  • Re-pilots: n≥2 conducted using same protocol as Task 1?
  • Honest: If metrics didn’t improve, explained why?

FAQs

Q: What if I can only recruit 3 participants? A: Minimum 4 required. Start earlier, use lab sessions to find peers. n=3 loses marks in Evidence Quality.

Q: Can I use mean instead of median? A: Median is more robust (less affected by outliers). Using mean is acceptable but justify why.

Q: What if my regression testing finds new issues? A: Fix them before re-pilots. That’s why regression testing exists. Document in notes.

Q: Do I need to implement screen reader fixes? A: Not mandatory, but WCAG violations affecting SR users are high inclusion impact (prioritise these).

Q: What if my fix breaks no-JS parity? A: That’s a fail in Fix Effectiveness. Test traditional path after every change.

Q: Can I get Excellent without Task 1→Task 2 evidence chains? A: No. Evidence chains are core to Evidence Chains criterion (20 marks). Every fix must link to Task 1 finding.

Q: How detailed should code diffs be? A: Show enough to understand the change (5-15 lines before/after). Full files not needed.

Q: What if I find no WCAG violations in Task 1? A: Unlikely if you tested with axe DevTools + keyboard + SR. Revisit Week 7 audit. If genuinely compliant, document that honestly.

Summary: What Gets High Marks?

✅ Evidence trails: Every claim traceable to data ✅ Method rigour: Consent, variants, systematic testing ✅ WCAG knowledge: Specific criteria cited, compliance verified ✅ Critical thinking: Systemic barriers analysed, not just “helps disabled people” ✅ Honest reporting: Limitations acknowledged, problems not hidden ✅ Process over product: Evaluation/redesign method matters, not code elegance

Focus on demonstrating HCI competence, not software engineering mastery.
