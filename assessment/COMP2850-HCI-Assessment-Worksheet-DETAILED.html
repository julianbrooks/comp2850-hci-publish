Assessment: Combined Evaluation & Redesign Portfolio

Academic Year 2025-26

This worksheet outlines the combined evaluation and redesign portfolio that forms your coursework assessment for COMP2850 HCI.

Portfolio submission: All evidence must be submitted via Gradescope by end of Week 10.

Expectations

Timeliness: You should complete all phases of the assessment in the order provided and submit your complete portfolio before the deadline (end of Week 10).

Presentation: You should present all your work clearly and concisely, following the submission template provided (submission-template.md). Evidence should be structured as CSV data, tables, code diffs, and screenshots (not essays).

Integrity: You are responsible for ensuring the evidence you submit is entirely your own work. You can find out more about academic integrity on the Skills@Library website. All work you submit for assessment is subject to the academic integrity policy. Specific requirements for this assessment:

  • No fabricated data: Pilot sessions must be genuine (verified by pilot notes)
  • No PII violations: Screenshots must be cropped/blurred, session IDs anonymous
  • Honest reporting: If fixes don‚Äôt improve metrics, analyse why (dishonest claims = fail)
  • AI tool citation: If using tools for code generation, cite in submission (e.g., ‚ÄúGitHub Copilot for route handlers‚Äù)

Feedback: Formative feedback will be provided during:

  • Week 10: Structured peer review (2 mins per portfolio, checklist-based)
  • Week 11: Class-wide feedback from staff in labs (identify gaps)

Support opportunities: Support is available via:

  • Lab classes (Weeks 9-11)
  • Module discussion forum
  • Office hours

Expected time for completion: 12-15 hours across Weeks 9-10.

Expected completion date: End of Week 10 (final deadline via Gradescope).

Assessment Summary

This coursework combines evaluation and redesign in a single evidence-based portfolio. You will:

  1. Evaluate your task management prototype through peer pilots (Week 9)
  2. Analyse data to identify usability and accessibility issues (Week 10)
  3. Redesign by implementing 1-3 fixes (Week 10)
  4. Verify improvements through regression testing and re-pilots (Week 10-11)

The assessment builds on:

  • Week 6: Needs-finding (job stories)
  • Weeks 7-8: Prototyping (task management interface, no-JS parity)
  • Week 9: Instrumentation (Logger.kt) and evaluation planning
  • Week 10: Data analysis, prioritisation, implementation, peer review
  • Week 11: Final refinement and submission

This is an evidence-based assessment: you‚Äôre graded on complete evidence chains (data ‚Üí finding ‚Üí fix ‚Üí verification), not essay quality. Total prose: ~100-200 words.

Assessment format: Pass/Fail outcome (40% threshold) with 0-100 grading for formative feedback:

  • 70-100: First (excellent) ‚Äî Complete evidence chains, rigorous WCAG verification
  • 60-69: 2:1 (very good) ‚Äî Strong evidence with minor gaps
  • 50-59: 2:2 (good) ‚Äî Adequate evidence with some issues
  • 40-49: 3rd (pass) ‚Äî Meets minimum standard
  • 0-39: Fail ‚Äî Below minimum OR fabrication OR PII violations

Code resources: Logger.kt starter code, submission template, and example portfolios are available on VLE/Gradescope.

Activity 1: Set up Logger.kt instrumentation (20 mins)

üî¥ Portfolio evidence required: Demonstrated Logger.kt generating metrics.csv during pilots

Task:

üî¥ Portfolio evidence required: submission-template.md Section 1 (Protocol & Tasks)

Task:

  2. Design 4-5 realistic evaluation tasks using the template:

Example task:

- **Action**: Add a new task with title "Buy milk before 6pm" to your list.
- **Success**: Task appears in list, no validation errors.
- **Target time**: <10 seconds
- **Linked to**: Week 6 Job Story #2 ("quickly add tasks without switching context")

Success criteria:

  • All tasks linked to Week 6 job stories
  • Success conditions are observable and measurable

üî¥ Portfolio evidence required: submission-template.md Section 4 (Implementation Diffs)

Example diff:

## Fix 1: Accessible Error Announcement

**Addresses finding**: Finding #1 (SR errors not announced)

**Before** (routes/TaskRoutes.kt:45):
```kotlin

call.respondText(error, ContentType.Text.Html)

After (routes/TaskRoutes.kt:45):

val error = """<span role="alert" aria-live="assertive">Title required</span>"""  // ‚úÖ ARIA
call.respondText(error, ContentType.Text.Html)

What changed: Added role="alert" and aria-live="assertive" to error span.

Why: Fixes WCAG 3.3.1 Error Identification (Level A). Errors now programmatically announced.

Impact: Screen reader users hear validation errors immediately. Reduces confusion, prevents multiple failed submissions. Benefits 2-5% of web users (WebAIM survey).


**Success criteria**:
- 1-3 fixes implemented
- All fixes documented with before/after diffs
- Each fix references specific finding from findings table
- Code changes tested and working

#### Activity 9: Regression testing (30 mins)

üî¥ **Portfolio evidence required**: submission-template.md Section 5 Part A (Verification Results - Regression Checklist)

**Task**:
1. Complete WCAG 2.2 AA regression checklist (20 checks) using Assistive Testing Checklist reference
2. Test categories:
   - **Keyboard** (5 checks): All features keyboard-accessible? Focus visible? No traps?
   - **Forms** (3 checks): Labels present? Errors identified? Name/role/value?
   - **Dynamic** (3 checks): Status messages? Live regions work? Focus management?
   - **No-JS** (3 checks): Feature parity? POST-Redirect-GET? Errors visible?
   - **Visual** (3 checks): Contrast minimum? Resize text? Non-text contrast?
   - **Semantic** (3 checks): Heading hierarchy? Bypass blocks? Alt text?
3. Mark each: `pass`, `fail`, or `n/a`
4. Document failures in notes column
5. Verify no Level A violations remaining

**Regression checklist format**:
```markdown
| Check | Criterion | Level | Result | Notes |
|-------|-----------|-------|--------|-------|
| K1 | 2.1.1 All actions keyboard accessible | A | pass | Tested Tab/Enter on all buttons |
| K2 | 2.4.7 Focus visible | AA | pass | 2px blue outline on all elements |
| F2 | 3.3.1 Errors identified | A | pass | Errors have role=alert (FIXED!) |

Success criteria:

  • All 20 checks completed
  • No Level A violations remaining (if any, must document why acceptable)
  • No-JS parity maintained

Academic Integrity Reminders

Allowed:

  • Using AI tools (ChatGPT, Claude, Copilot) for code generation (must cite in submission)
