COMP2850 HCI: Combined Evaluation & Redesign Assessment

Academic Year: 2025-26 Assessment Type: Coursework (Pass/Fail with 0-100 grading) Timeline: Weeks 9-11 (Launch Week 9, Due end Week 10) Pass Threshold: 40/100 marks

Overview

What: Evidence-based portfolio combining evaluation and redesign in a single submission

Why: Demonstrate HCI competence through rigorous evaluation → data-driven redesign → verified improvement

How: Run peer pilots (n=2 min), collect metrics, implement accessibility fixes, verify with regression testing

Format: Evidence artifacts (CSV, tables, code diffs, screenshots) + minimal prose (~100-200 words)

Deliverables (6 Files + Evidence Folder)

Students submit ONE complete portfolio containing:

1. protocol-tasks.md (Bullet format, minimal prose)

Content:

  • 4-5 evaluation tasks (scenario, action, success, target time)
  • Consent script (verbatim, with rights/withdrawal)
  • Link to Week 6 job stories (LO2 evidence)

Format: Bullet lists, no essay paragraphs

Example:

## Task 3: Add Task
- **Scenario**: Track new deadline "Buy milk"
- **Action**: Add task with title "Buy milk"
- **Success**: Task appears in list, no errors
- **Target**: <10 seconds
- **Linked to**: Week 6 Job Story #2 ("quickly add tasks")

Word count: ~50-100 words (just scenario context)

2. findings-table.csv (Evidence chains as structured data)

Content: Each finding as a row with evidence sources

Columns:

finding,data_source,observation,wcag,impact,inclusion,effort,priority

Formula: Priority = (Impact + Inclusion) - Effort

3. metrics.csv (Raw pilot data from Logger.kt)

Content: Server-side instrumentation output

Format: Standard Week 9 Lab 1 schema

ts_iso,session_id,request_id,task_code,step,outcome,ms,http_status,js_mode

  • n=2 minimum participants (n=4 ideal)
  • Anonymous session IDs (P1_xxxx, P2_xxxx format)

4. implementation-diffs.md (Before/after code, minimal narrative)

Content: 1-3 fixes with code changes

Format: Before snippet → After snippet → Impact statement

Example:

## Fix 1: Accessible Error Announcement

**Before** (routes/TaskRoutes.kt:45):
```kotlin
val error = "Title required"  // ❌ No ARIA
call.respondText(error, ContentType.Text.Html)

After (routes/TaskRoutes.kt:45):

val error = """<span role="alert" aria-live="assertive">Title required</span>"""  // ✅ ARIA
call.respondText(error, ContentType.Text.Html)

Impact: SR users now hear validation errors (WCAG 3.3.1 Level A pass) Verification: verification.csv row 12 shows pass

[1-3 fixes total]


**Word count**: ~50-100 words (just "Impact" statements)

Participant Requirements (Flexible)

Evaluation Phase (Week 9):

  • Minimum: n=2 peer pilots
  • Ideal: n=4 peer pilots
  • Variant requirement: At least 1 non-standard variant (keyboard-only OR screen reader OR no-JS)

Verification Phase (Week 10):

  • Minimum: n=1 re-pilot (must use appropriate variant for fix — e.g., SR user if fixing SR issue)
  • Ideal: n=2 re-pilots (better comparison)

Rationale:

  • n=2 shows pattern (not anecdotal)
  • Flexible for scheduling/availability
  • Quality over quantity (1 SR user > 4 mouse users if fixing SR accessibility)

Pass Criteria (Must Meet ALL)

✅ Evidence Complete:

  • metrics.csv present with n=2+ participants
  • findings-table.csv has 3+ findings with evidence sources
  • verification.csv has all 20 regression checks completed
  • implementation-diffs.md shows 1-3 code changes

✅ Evidence Chains:

  • findings-table.csv “data_source” column links to metrics.csv line numbers OR pilot notes timestamps
  • implementation-diffs.md references findings from findings-table.csv
  • verification.csv shows before/after metrics for implemented fixes

✅ WCAG Compliance:

  • verification.csv shows no Level A violations remaining
  • At least 1 WCAG criterion improved (fail → pass)

✅ Professional Practice:

  • No PII violations (screenshots cropped, session IDs anonymous)
  • Honest reporting (if fix didn’t improve metrics, limitation acknowledged in verification.csv notes)
  • No-JS parity maintained (verification.csv shows both paths functional)

Fail Triggers (Any ONE Causes Fail)

❌ Fabrication (automatic 0 marks):

  • Data too perfect (all success, no errors, unrealistic times)
  • No pilot notes (pilots didn’t happen)
  • Metrics contradict claims (e.g., findings say n=4 but CSV shows n=2)

❌ Severe PII Violations (-10 marks + ethics fail):

  • Names, emails, student IDs visible in screenshots
  • Identifiable session IDs (real names used)
  • Participant consent not documented

❌ Major Gaps (0 marks for missing section):

  • n<2 participants OR no metrics.csv
  • No regression checklist OR <10 checks completed
  • No redesign implemented OR no code diffs shown
  • No before/after comparison

Peer Review (Week 10 Lab 2)

Format: In-lab structured session, 2 mins per peer

Checklist:

## Evidence Check (2 mins)
- [ ] ✅ / ❌ — metrics.csv present and opens
- [ ] ✅ / ❌ — findings-table.csv has data (3+ rows)
- [ ] ✅ / ❌ — verification.csv complete (20 checks)
- [ ] ✅ / ❌ — implementation-diffs.md shows code changes
- [ ] ✅ / ❌ — No PII visible in screenshots

**On track to pass?** Yes / No / Borderline
**If No/Borderline, missing:** ___________

Process:

  1. Form groups of 4 (20 groups per 80-student lab)
  2. Each student reviews 2 peers (4 mins total)
  3. Give structured feedback (1 min per peer)
  4. Staff circulate, answer questions, flag at-risk students

Staff Marking (5 Minutes Per Portfolio)

Decision tree:

  1. Quick checks (1 min):

    • 6 files present?
    • CSV files open?
    • Spot-check 2-3 screenshots for PII

  2. Pass criteria (2 mins):

    • metrics.csv: n=2+? Columns correct?
    • findings-table.csv: Links to data sources?
    • verification.csv: 20 checks complete? WCAG Level A pass?
    • implementation-diffs.md: Code changes shown?

  3. Fail triggers (1 min):

    • Fabrication indicators? (too perfect, no notes, contradictions)
    • PII violations? (checked in step 1)
    • Major gaps? (n<2, no regression, no redesign)

  4. Assign mark (1 min):

    • 70-100: Excellent (complete, rigorous, professional)
    • 60-69: Very good (strong with minor gaps)
    • 50-59: Good (adequate with issues)
    • 40-49: Pass (meets minimum)
    • 0-39: Fail (below minimum OR trigger present)

Common Questions

Q: Can I use n=2 participants instead of n=4? A: Yes. n=2 is minimum acceptable. n=4 is ideal (shows stronger patterns) but not required.

Q: Do I need to write essays? A: No. This is evidence-based. Total prose: ~100-200 words (task scenarios, impact statements, alt text). Rest is CSV/tables/code.

Q: What if my fix doesn’t improve metrics? A: Document honestly in verification.csv notes column. Analyse why (sample size? wrong fix? needs iteration?). Honesty valued over perfect results.

Q: Can I pilot with friends? A: Yes, if they consent. Peer pilots are ideal. Just document consent process.

Q: What if I can’t find participants? A: Post in module discussion forum. Coordinate with lab peers. TAs can help connect students needing participants.

Q: Can I use AI tools (e.g., ChatGPT, Claude, Copilot)? A: Yes for code generation (must cite: “Generated with X”). No for analysis/findings (must be your own thinking).
